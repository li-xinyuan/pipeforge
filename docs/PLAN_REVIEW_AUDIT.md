## 三、阻断性问题

### 🔴 B-01：`get_current_user` 架构假设错误 — T-5A-04 方案不可行

**计划声称**（T-5A-04 第 117 行）：
```python
def require_role(*roles: str):
    def dependency(user: User = Depends(get_current_user)):
        ...
```

**代码库事实**：`get_current_user` 定义在 `configforge/api/auth.py:111`，是一个 **FastAPI 路由处理函数**（`@router.get("/me")`），而非 FastAPI 依赖注入函数。当前代码库中不存在可用于 `Depends()` 的认证依赖。

**当前认证机制**：`configforge/middleware/auth.py` 中的 `AuthMiddleware` 是 `BaseHTTPMiddleware` 子类，通过 `request.state.user` 传递用户信息，而非 FastAPI 的依赖注入体系。

**影响**：T-5A-04 和 T-5A-05 的方案需要彻底重写。正确做法是：
1. 从 `AuthMiddleware` 中提取 `request.state.user` 的 FastAPI 依赖包装函数
2. 或使用 `fastapi.Depends` + 自定义 `Security` 依赖读取 `request.state`

**建议修正**：
```python
# configforge/middleware/auth.py 新增
from fastapi import Request, HTTPException, Depends

async def get_current_user(request: Request) -> User:
    user = getattr(request.state, 'user', None)
    if user is None:
        raise HTTPException(401, detail="未认证")
    return user

def require_role(*roles: str):
    async def dependency(request: Request, user: User = Depends(get_current_user)):
        if user.role not in roles:
            raise HTTPException(403, detail="权限不足")
        return user
    return dependency
```

### 🔴 B-02：`CONFIGFORGE_ENV` 环境变量不存在 — T-5A-06 缺少前置步骤

**计划声称**（T-5A-06 第 205 行）：在 `server.py` 启动时检查 `CONFIGFORGE_ENV=production`

**代码库事实**：在整个后端代码库中，`CONFIGFORGE_ENV` 和 `NODE_ENV` **均不存在**。`server.py` 中没有任何环境区分逻辑。Recommendations 文档 O-05 中引用的是 `NODE_ENV`（前端惯例），而 Plan 中使用的是 `CONFIGFORGE_ENV`（Python 惯例），两文档术语也不一致。

**影响**：T-5A-06 隐含了一个前置任务——首先需要在 `server.py` 中引入 `CONFIGFORGE_ENV` 环境变量的读取和模式判断逻辑。此外，T-5A-02（Fernet 检测）中也引用了 `CONFIGFORGE_ENV=production`，同样受影响。

**建议修正**：在 Phase 5A 增加一个前置任务或扩展 T-5A-06 的范围，明确包含：
1. `server.py` 新增 `CONFIGFORGE_ENV` 读取逻辑（默认 `development`）
2. 文档统一术语为 `CONFIGFORGE_ENV`

### 🟡 I-05：T-5B-04 手风琴模式可能与现有滚动对齐冲突

当前 `ConfigWizardView.vue` 使用 **固定 170px 滚动步长**（参考 commit `3623623`）来对齐步骤切换。手风琴模式会动态改变各步骤卡片的高度，可能破坏滚动对齐逻辑。

**建议**：在 T-5B-04 方案中增加「适配滚动对齐逻辑」子任务。

### 🟡 I-06：T-5B-08 API 错误处理方案是 Breaking Change

Plan 决定将 `useApi.ts` 的 `request()` 从「返回 null」改为「抛出 `ApiError`」，这意味着所有调用 `request()` 的地方（约 10+ 个 composable 和组件）需要同步修改错误处理逻辑。Plan 中「所有调用方改为 try/catch 处理」只有一句话，但这是跨文件的重构。

**建议**：
- 在 Phase 5B 中为 T-5B-08 单独增加工作量估算（建议 2-3 天）
- 或考虑渐进式方案：新增 `requestOrThrow()` 方法，逐步迁移调用方

---

### 🟡 I-07：T-5E-05 插件系统范围过大

「插件市场：社区贡献的插件可安装」隐含了插件注册中心、版本管理、安全审核、CLI 工具等一系列基础设施。这远超 v1.2.0 的合理范围。

**建议**：将 T-5E-05 拆分为两个任务：
- T-5E-05a（v1.2.0）：本地插件接口 + 自动加载机制
- T-5E-05b（v2.0.0）：插件市场 + 远程安装

## 八、总体评价

- 从安全→性能→代码质量→功能→架构的分层递进顺序正确

- 工作量和依赖关系可以更精确（避免执行中反复调整）
- 部分任务范围需要收窄或分期（插件市场、i18n）

**下一步行动**：
1. 修复 3 个阻断性问题（预计 0.5 天）
2. 评估 7 个重要问题的修复优先级
