## 二、A — 多步 Pipeline

### 2.1 Processor 类型

**两种计算范式，穷举了 SQLite 批处理架构下所有表→表转换：**

| Processor | 范式 | 说明 |
|-----------|------|------|
| SQL | 声明式 / 集合操作 | 已有。SELECT / WHERE / JOIN / GROUP BY / 子查询 |
| Python | 命令式 / 逐行过程 | 新增。正则、外部 API、复杂计算、文件读写 |

> Filter、Join/Merge 是 SQL 的子集，不做独立类型。
> Template 归入 Output 层扩展，Branch 归入编排层（非 v0.3 必需）。
> Lookup、AI Transform 是 Python 的便捷封装（零代码配置），后续版本实现。

### 2.2 YAML 模型

```yaml
processors:
  - name: 数据清洗
    plugin: python
    input_tables: [person_raw, attendance_raw]
    output_tables: [clean_data]
    config:
      script: |
        def process(ctx):
            ctx.db.execute("DELETE FROM person_raw WHERE name IS NULL")
            ctx.db.execute("""
                CREATE TABLE clean_data AS
                SELECT p.*, a.date, a.status
                FROM person_raw p
                JOIN attendance_raw a ON p.id = a.person_id
            """)

  - name: 出勤统计
    plugin: sql
    input_tables: [clean_data]
    output_tables: [monthly_report]
    config:
      sql: |
        SELECT dept AS "部门", COUNT(*) AS "出勤天数"
        FROM clean_data
        WHERE date >= '{{month_start}}'
        GROUP BY dept
```

**关键设计：**
- `input_tables` — 显式声明依赖，替代隐式顺序
- `output_tables` — 声明产出表名
- 引擎按 DAG 拓扑排序执行，自动检测循环依赖
- Jinja2 变量 `{{param}}` 继续支持

### 2.3 引擎改动

**PipeForge `engine.py`：**
- 移除 `processors[:1]` 限制
- 新增拓扑排序（基于 `input_tables` / `output_tables` 依赖关系）
- 每步执行前检查 `input_tables` 是否存在
- 支持并行执行无依赖关系的步骤

**ConfigForge `pipeline.py`：**
- `execute_pipeline` 传递完整 processor 列表
- 错误时标记失败的 processor，保留已完成步骤的中间表
- 返回 `failed_step: int` 和 `intermediate_tables: list[str]`

### 2.4 前端 Step 3 改造

**从单 SQL 编辑器 → Processor 链式列表：**

- 每个 processor 一张卡片，显示：类型标签（SQL/Python）、名称、输入表、输出表、代码
- 「+ SQL 步骤」「+ Python 步骤」按钮添加新卡片
- 拖动排序（依赖关系自动重算）
- 展开/折叠卡片编辑区域
- Dry-run 按钮：单独执行当前步骤查看中间结果
- 执行整个链：右上角「试运行并下载」

## 三、G — AI 自动编排

### 3.1 两种模式

| 模式 | 用户 | 交互 |
|------|------|------|
| 手动定义 | 懂技术的人 | 点击「+ SQL/Python」→ 填写配置 → 拖动排序 |
| AI 编排 | 不写代码的人 | 自然语言描述 → AI 规划链 → 审核 → 填入向导 |

两种模式不是二选一——**用户可以 AI 生成框架后手动修改，也可以手动搭完再让 AI 优化某一步。**

### 3.2 AI 编排流程

1. 用户在 Chat Panel 用自然语言描述最终需求
2. AI 读取当前输入源信息（表名、列名、样本数据）作为上下文
3. AI 返回多步处理计划（JSON 格式）：
   - 每个 step：`type`（sql/python）、`name`、`input_tables`、`output_tables`、`code`
4. Chat Panel 渲染计划：每步一张卡片，代码高亮
5. 用户可：确认全部、修改某步、删除某步、重新生成
6. 确认后一键填入 Step 3 向导，变成普通 Processor 卡片

### 3.3 AI 上下文

```json
{
  "category": "orchestrate",
  "inputs": [
    {"table": "person", "columns": ["id", "name", "dept"], "sample": [...]},
    {"table": "attendance", "columns": ["person_id", "date", "status"], "sample": [...]}
  ],
  "outputColumns": ["部门", "出勤天数", "出勤率"],
  "naturalLanguage": "统计各部门本月的出勤率"
}
```

### 3.4 AI 返回格式

```json
{
  "steps": [
    {"type": "python", "name": "数据清洗", "input_tables": ["person", "attendance"], "output_tables": ["clean"], "code": "..."},
    {"type": "sql", "name": "出勤统计", "input_tables": ["clean"], "output_tables": ["report"], "code": "..."}
  ],
  "explanation": "第一步用 Python 清洗合并数据，第二步用 SQL 聚合统计"
}
```

### 3.5 Prompt 设计

新增 `orchestrate` 类别：

```
你是一个数据流水线架构师。用户提供输入源（表名+列名+样本数据）和输出目标，你需要规划处理链。

规则：
1. 每步声明 input_tables 和 output_tables
2. 选择合适的 processor 类型：sql（查询/过滤/聚合）或 python（清洗/外部调用/复杂逻辑）
3. SQL 步骤用 SQLite 语法，表名用双引号包裹
4. Python 步骤用 def process(ctx) 函数签名
5. 步骤数 ≤ 5，尽量简洁
```

## 四、B — 错误体验

### 4.1 分层处理

| 层级 | 错误类型 | 处理方式 |
|------|---------|---------|
| 前端 | 网络断开/超时 | 中文提示 + 重试按钮 |
| API | 文件格式/SQL 错误/参数校验 | 中文消息 + 修复建议 |
| 引擎 | Pipeline 执行失败 | 标记失败步骤 + 保留中间结果 |

### 4.2 API 错误映射

```python
# 当前 vs 目标
"sqlite3.OperationalError: no such column: dept_name"
# → "SQL 错误：列 'dept_name' 不存在。可用列：dept, name, count。"

"openpyxl.utils.exceptions.InvalidFileException"
# → "文件格式不支持：请上传 .xlsx 或 .csv 格式的文件。"

"UnicodeDecodeError: 'utf-8' codec can't decode"
# → "CSV 文件编码不是 UTF-8，请尝试选择 GBK 编码或转换文件后再上传。"
```

### 4.3 引擎错误

Pipeline 执行失败时返回：

```json
{
  "error": "Step 2 '过滤本月' 执行失败",
  "failed_step": 2,
  "intermediate_tables": ["clean_data"],
  "detail": "SQL 错误：列 'date' 不存在。可用列：id, name, dept, attendance_date。"
}
```

前端可展示：失败位置、已完成步骤、中间结果可预览。
