## 二、前置修复 — `execute_dry_run` bug

**现状:** `pipeline.py` L247 调用 `engine.execute_dry_run(params, log_dir=LOG_DIR)`，但 `PipelineEngine` 未定义此方法，前端 Step 3「试运行」按钮必然报 `AttributeError`。

**修复:** 在 `PipelineEngine` 中实现 `execute_dry_run` 方法——执行 input + processor 阶段，跳过 output，返回中间表数据供前端预览。

**优先级:** v0.3.0 任何功能开发前先修此 bug。

## 三、A — 多步 SQL Pipeline（v0.3.0）

### 3.1 Processor 类型规划

| 版本 | 可用类型 |
|------|---------|
| v0.3.0 | **SQL**（多步链式） |
| v0.4.0 | SQL + **Python**（沙箱隔离后引入） |

> v0.3.0 只实现 SQL 多步链式。Filter、Join/Merge 是 SQL 的子集，不做独立类型。

### 3.2 YAML 模型

```yaml
processors:
  - name: 过滤本月数据
    plugin: sql
    input_tables: [raw_data]
    output_tables: [this_month]
    config:
      sql: |
        CREATE TABLE this_month AS
        SELECT * FROM raw_data
        WHERE date >= '{{month_start}}'

  - name: 出勤统计
    plugin: sql
    input_tables: [this_month]
    output_tables: [monthly_report]
    config:
      sql: |
        SELECT dept AS "部门", COUNT(*) AS "出勤天数"
        FROM this_month
        GROUP BY dept
```

**关键设计:**
- `input_tables` — 显式声明依赖，完全替代隐式「从上一步读」的假定
- `output_tables` — PipelineEngine 的 `ProcessorSpec` 已有此字段（`models.py` L108），v0.3 复用
- 引擎按 DAG 拓扑排序执行，Kahn's algorithm 检测循环依赖
- Jinja2 变量 `{{param}}` 继续支持

### 3.3 引擎改动

**PipeForge `engine.py`:**

`PipelineEngine._execute_processor` 需增加前置检查：
```python
for table in proc_spec.input_tables:
    if table not in context.db.list_tables():
        raise ValueError(f"Input table '{table}' not found")
```

拓扑排序逻辑（伪代码）：
```python
def _topological_sort(processors):
    available = set(context.db.list_tables())  # 初始表来自 input plugins
    ordered = []
    remaining = list(processors)
    while remaining:
        ready = [p for p in remaining if all(t in available for t in p.input_tables)]
        if not ready:
            raise ValueError("Circular dependency or missing input table")
        for p in ready:
            ordered.append(p)
            available.update(p.output_tables)
            remaining.remove(p)
    return ordered
```

支持并行执行无依赖关系的步骤（`asyncio.gather`）。

**ConfigForge `pipeline.py`:**

- `execute_pipeline` / `dry_run` 传递完整 processor 列表
- 错误时标记失败的 processor index，保留已完成步骤的中间表（不清理 SQLite）
- 返回 `failed_step: int | None` 和 `intermediate_tables: list[str]`

### 3.4 前端 Step 3 改造

**从单 SQL 编辑器 → Processor 列表（添加/删除/编辑，v0.3.0 不做拖拽）：**

- 每个 SQL processor 一张卡片，显示：名称、输入表下拉、输出表名、SQL 代码
- 「+ SQL 步骤」按钮添加新卡片
- × 按钮删除卡片
- 展开/折叠卡片编辑区域
- Dry-run 按钮：单独执行当前步骤查看中间结果
- 执行整个链：「试运行并下载」按钮

### 3.6 对 Step 4（输出配置）的影响

当前 `sourceTable` 选项来自单个 processor 的 `outputTable`。多步 Pipeline 后，**`sourceTable` 需要从所有 processor 的 `output_tables` 中选取**。

`OutputConfigTab.vue` 改动：
```typescript
// 旧：单表
const sourceTables = computed(() => [store.processor.outputTable])

// 新：所有 processor 产出表
const sourceTables = computed(() =>
  store.processors.flatMap(p => p.outputTables)
)
```

## 四、B — 错误体验（v0.3.0 第一层 + v0.3.1 第二层）

# configforge/utils/error_messages.py（新建）
ERROR_MESSAGES = {

## 五、G — AI SQL 步骤链编排（v0.3.1）

### 5.1 简化策略

评审指出的 4 个风险（JSON 可靠性、Python 代码安全、上下文窗口、审核复杂度）完全成立。v0.3.1 AI 编排做以下简化：

1. **只生成 SQL 步骤**，不生成 Python（Python 推迟到 v0.4）
2. **上下文精简**：只传表名+列名，不传样本数据；SQL 生成后用户可单独对某步调用 AI 优化
3. **MVP 交互简化**：只做"确认全部 / 重新生成"，不做单步修改（依赖关系断裂风险）
4. **独立确认**：每步可单独试运行，确认通过后再进入下一步

### 5.2 编排流程

1. 用户在 Chat Panel 用自然语言描述最终需求
2. AI 读取输入源信息（仅表名+列名）和目标输出列
3. AI 返回 SQL 步骤链：
```json
{
  "steps": [
    {"name": "过滤本月", "input_tables": ["person", "attendance"], "output_tables": ["this_month"], "sql": "CREATE TABLE this_month AS ..."},
    {"name": "出勤统计", "input_tables": ["this_month"], "output_tables": ["report"], "sql": "SELECT ..."}
  ],
  "explanation": "第一步过滤本月数据，第二步聚合统计"
}
```
4. Chat Panel 渲染步骤列表
5. 用户「确认全部」→ 一键填入 Step 3 向导，变成普通 Processor 卡片
6. 用户可手动调整每步的 SQL 和 input/output tables

### 5.3 降级策略

AI 返回格式错误时的处理：
1. JSON 解析失败 → 展示 AI 原始文本，引导用户重试
2. 缺少必要字段（name/input_tables/output_tables/sql）→ 跳过不合规步骤，展示可用的
3. 完全无法解析 → 降级为单步 SQL 生成（回退到当前 chat 模式）
4. 连续 3 次失败 → 建议用户切换为手动模式

### 5.4 Prompt 设计

新增 `orchestrate` 类别：

```
你是一个数据流水线架构师。用户提供输入源和输出目标，规划 SQL 步骤链。

上下文：
- 输入表及其列名（名称、类型）
- 目标输出列
- 用户的自然语言需求

规则：
- 每步必须声明 input_tables（依赖哪些表）和 output_tables（产出哪些表）
- 只能使用 SQLite 语法，表名和列名用双引号包裹
- 不要编造列名——只能使用上下文中给出的列名
- 步骤数 ≤ 5，尽量简洁

返回纯 JSON（不要包裹在 markdown 代码块中）：
{"steps": [{"name": "...", "input_tables": [...], "output_tables": [...], "sql": "..."}], "explanation": "..."}
```

## 六、向后兼容方案

v0.3 将 `processor` 从单对象改为数组，需兼容旧数据：

### 6.1 已保存配置（`configs/` 目录）

`loadFromConfigState` 检测旧格式并自动升级：

```typescript
function loadFromConfigState(stateDict: any) {
  // 兼容旧格式：processor 为对象 → 包装为数组
  const rawProcessors = stateDict.processors || stateDict.processor
  if (rawProcessors && !Array.isArray(rawProcessors)) {
    processors.value = [migrateProcessor(rawProcessors)]
  } else {
    processors.value = (rawProcessors || []).map(migrateProcessor)
  }
}
```

### 6.2 localStorage 持久化

Pinia persist key 从 `wizard_state_v1` 升级为 `wizard_state_v2`。

### 6.3 API 层

`/api/configs/{id}` 加载时，后端检测旧格式并自动转换：
```python
if "processor" in state and "processors" not in state:
    state["processors"] = [state.pop("processor")]
```

### 6.4 前端/后端类型对齐

当前前端 `ProcessorConfig` 用 `outputTable: string`（单数），后端 `ProcessorSpec` 用 `output_tables: list[str]`。v0.3 统一为 `outputTables: string[]`（前端）/ `output_tables: list[str]`（后端）。
