## 一、整体评价

**这是一份成熟的开源项目路线图。** 最大的亮点是 §7「不做的事情」——明确排除比明确纳入更难，说明设计者对项目边界有清醒认知。§5 与 NEXT_STEPS.md 的差异对比也很有价值，展示了思考演进过程。

### 做得好的地方

1. **定位精准**：「让不懂代码的人也能创建可重复的数据处理流程」——一句话说清了和 dbt/Airflow 的区别
2. **三阶段递进合理**：降低门槛 → 扩展能力 → 社区效应，每阶段有明确的核心目标
3. **架构变更提前规划**：§4 每阶段的关键架构变更都有代码示例，不是空谈
4. **Plugin SDK 优先级提升**：从 NEXT_STEPS 的 P2 提升到 P0，这是正确的——没有 SDK 就没有社区
5. **成功指标可量化**：100+ stars、5+ PR、3+ 社区插件——可衡量
6. **§7 不做的事情**：数据血缘、SSO、自建调度器等明确排除，避免范围蔓延
7. **§5 差异对比**：与 NEXT_STEPS.md 的 7 处差异都有理由，展示了决策思考

## 二、建议调整的地方

### P1 — 重要建议

#### P1-1: Phase 1 缺少 ConfigForge 测试补充

NEXT_STEPS.md 将「ConfigForge 测试」列为 Phase 1 🟡 优先级。当前 ConfigForge 前端几乎没有测试（只有 store 测试），后端 26 个测试未验证。路线图 §2 提到「26 个测试」「12 个 E2E 场景」，但如果这些测试不可靠，开源后会很被动。

**建议**：在 Phase 1 表格中增加一行：

| 🟡 P1 | **ConfigForge 测试加固** | ConfigForge | 后端测试全部通过 + 前端关键路径 Vitest 测试 |

---

#### P1-2: Phase 2 的 Cron 调度与「不自建调度器」原则的边界需要更清晰

§7 明确说「不自建调度器」，Phase 2 说「集成系统 cron」。但 `pipeforge schedule` 命令本身就是一个调度管理层——它需要：
- 管理 schedule 列表（增删改查）
- 持久化 schedule 配置
- 处理时区
- 错误处理（cron 执行失败怎么办？）

这实际上已经是一个轻量调度器了。

**建议**：Phase 2 的 Cron 调度限定为：
- `pipeforge schedule add/list/remove` — 管理系统 crontab 条目
- 不做执行历史、重试、告警
- 不做 Web UI 管理
- 在路线图中明确标注这些边界

---

### P2 — 建议优化

#### P2-1: Phase 1 交付物中缺少 CLI 完善项

开源项目第一印象很重要。当前 CLI 的 `--help` 信息是否完整？`--version` 是否可用？这些是 5 分钟跑通 demo 的前提。

**建议**：在 Phase 1 交付物中增加：
- `pipeforge --version` 可用
- `pipeforge --help` 信息完整（含示例）
- `pipeforge init` 交互式创建配置（可选）

---

#### P2-2: 成功指标缺少「技术指标」

当前指标都是社区指标（stars/PR/模板数），建议补充技术指标：

| 阶段 | 技术指标 |
|------|---------|
| Phase 1 | PipeForge 测试覆盖率 ≥80%，ConfigForge 后端测试全部通过 |
| Phase 2 | 插件安装到可用 < 10 分钟，SDK 文档完整度 100% |
| Phase 3 | REST API 响应时间 < 500ms，模板市场搜索 < 1s |

---

#### P2-3: §2 ConfigForge 状态中「26 个测试」「12 个 E2E 场景」需验证

这些数字在 FULL_PROJECT_SCAN.md 中标记为「未验证」。如果实际测试不可运行，写在路线图中会误导读者。

**建议**：验证后标注实际数字，或改为「目标：26 个后端测试 + 12 个 E2E 场景」。

---

#### P2-4: Phase 3 的「配置模板市场」缺少技术方案

Phase 1/2 的关键架构变更都有代码示例，但 Phase 3 只有目录结构，缺少：
- 模板如何上传/下载？（Git 仓库？API？）
- 模板元数据 schema？
- 模板适配检测（用户 Excel 是否匹配模板输入）？

**建议**：补充最小可用版的技术方案，至少明确：
- 存储：Git 仓库（monorepo）vs API + 数据库
- 分发：`pipeforge template search/install` CLI 命令
- 适配：上传 Excel 后自动匹配模板的列名相似度算法

---

### P3 — 微小问题

#### P3-1: §4 Phase 1 Jinja2 示例中 SQL 引号不一致

```yaml
# 之前
sql: "SELECT * FROM person WHERE month = '2026-05'"

# 之后
sql: "SELECT * FROM person WHERE month = '{{target_month}}'"
```

之前用单引号包裹字符串值 `'2026-05'`，之后模板变量没有引号 `'{{target_month}}'`。Jinja2 渲染后会是 `month = 2026-05`（无引号），SQL 会报错。应该是 `month = '{{target_month}}'`。

---

#### P3-2: §6 里程碑时间线偏紧

2026 Q2 现在已经快过半了（5 月），Phase 1 目标内容不少（CSV 入/出 + Jinja2 + AI SQL + Dry-Run + 开源门面），2 个月内完成有压力。

**建议**：标注「Phase 1 预计 2-3 个月」，或调整为「2026 Q2-Q3」。
