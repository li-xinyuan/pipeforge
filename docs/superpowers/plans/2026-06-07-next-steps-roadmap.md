# ConfigForge 下一步建设方向与工作计划

> 日期：2026-06-07
> 基于：项目现状全面分析 + 用户需求推演 + 技术债务梳理

---

## 一、现状定位

ConfigForge 当前处于 **v0.4 阶段**：核心的"Excel/CSV → SQL 处理 → Excel/CSV 输出"流程基本可用，检查点规则、版本管理、执行历史已实现。但距离生产可用还有显著差距。

**核心矛盾**：系统更像"YAML 配置生成器"而非"数据管道平台"。用户配置完管道后，只能手动触发执行一次，无法定时运行、无法监控状态、无法自动告警。

---

## 二、下一步建设方向（4 条路线）

### 路线 A：补齐基础 → 让现有功能真正可用（优先级最高）

当前有多个"硬阻塞"问题导致核心流程走不通，必须先修。

### 路线 B：从工具到平台 → 定时调度 + 执行监控

这是产品定位升级的关键跳板。没有调度和监控，ConfigForge 永远只是个配置工具。

### 路线 C：扩展数据能力 → 更多输入输出 + 数据库直连

用户最常见的场景是"数据库 A 的数据 → 处理 → 写入数据库 B"，当前 Database 输出对 MySQL/PG 无效。

### 路线 D：降低使用门槛 → 模板市场 + 新手引导 + AI 增强

让非技术用户也能快速上手。

## 五、关键决策点

### 决策 1：定时调度的技术方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: APScheduler（进程内）** | 简单，无额外依赖 | 单进程，重启丢任务 |
| **B: Celery + Redis/RabbitMQ** | 分布式，可靠 | 架构复杂，运维成本高 |
| **C: 系统级 Cron** | 最简单 | 不跨平台，不好管理 |

**建议**：v1.0 用方案 A（APScheduler），v2.0 再考虑 B。

### 决策 2：Database 输出的实现策略

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A: SQLAlchemy Engine 直连** | 通用，支持所有数据库 | 依赖 SQLAlchemy，连接管理复杂 |
| **B: 每种数据库独立插件** | 灵活，可按需安装 | 代码重复，维护成本高 |

**建议**：方案 A，复用现有 ConnectionStore 的连接配置。

### 决策 3：AI 引导是否继续推进

当前 AI 引导功能已移除，改为固定提示。是否重新引入？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **A: 暂不引入，专注基础** | 减少复杂度，先做好核心 | 用户体验上限低 |
| **B: 重新引入简化版** | SQL 生成 + 列映射是刚需 | AI 不可用时体验差 |
| **C: AI 作为可选增强** | 不影响基础流程 | 需要两套 UI 逻辑 |

**建议**：选项 C。保持当前固定提示为基础，AI 作为右侧面板的可选增强（类似之前的 AiChatPanel sidebar 模式），AI 不可用时面板显示"前往设置配置 AI"。

## 八、风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| APScheduler 单进程不够用 | 中 | 定时任务丢失 | 监控进程健康，自动重启 |
| Database 输出连接池泄漏 | 高 | 数据库连接耗尽 | 连接池大小限制 + 超时回收 |
| AI API 费用过高 | 低 | 用户不愿使用 | 本地模型 fallback + 缓存 |
| 中文列名兼容性引发新 bug | 中 | 数据处理错误 | 充分测试 + 渐进式发布 |
