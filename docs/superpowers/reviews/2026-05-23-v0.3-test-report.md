### 2.4 向后兼容 ⚠️ 部分失败

| # | 测试项 | 结果 | 详情 |
|---|--------|------|------|
| 1 | 保存旧格式配置 | ✅ | status=200 |
| 2 | 加载旧格式配置 | ✅ | status=200 |
| 3 | **旧格式迁移到 processors[]** | ❌ | processors 为空数组，迁移未生效 |
| 4 | 从旧格式生成 YAML | ✅ | YAML 正确包含 processors 节 |

**Bug 详情**:

`_migrate_state_dict()` 的迁移条件为 `"processor" in state_dict and "processors" not in state_dict`，但前端保存旧格式时同时发送了 `processors: []` 和 `processor: {...}`，导致 `processors` key 已存在，迁移条件不满足。

**根因**: [configs.py:27](file:///Users/lixinyuan/code/CCTEST/configforge/api/configs.py#L27)

```python
# 当前代码（有 bug）
if "processor" in state_dict and "processors" not in state_dict:
    proc = state_dict.pop("processor")
    state_dict["processors"] = [proc]

# 应改为
if "processor" in state_dict and not state_dict.get("processors"):
    proc = state_dict.pop("processor")
    if "outputTable" in proc:
        proc["output_tables"] = [proc.pop("outputTable")]
    state_dict["processors"] = [proc]
```

## 三、前端 UI 测试详情

### 3.1 🔴 严重问题：locked 状态 StepCard 阻塞按钮点击

**现象**: 当 StepCard 处于 `locked` 状态时，其 `pointer-events: none` 样式导致内部所有按钮无法被物理点击。`wizard__steps` div 成为 `elementFromPoint` 返回的顶层元素。

**影响范围**:
- "+ SQL 步骤" 按钮（Step 3 locked 时）
- "保存并继续 ↓" 按钮（Step 2/3/4 locked 时）
- "▶ 运行查询" 按钮（Step 3 locked 时）
- "保存配置" 按钮（Step 5 locked 时）

**Playwright 测试证据**:
```
Button "+ SQL 步骤": blocked=true, topElement="wizard__steps"
Button "保存并继续 ↓": blocked=true, topElement="wizard__steps"
Button "保存配置": blocked=true, topElement="wizard__steps"
```

**根因**: [WizardStepCard.vue:60](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/wizard/WizardStepCard.vue#L60)

```css
.wizard-step-card--locked {
  opacity: 0.55;
  pointer-events: none;  /* ← 这导致所有子元素不可点击 */
}
```

**修复建议**: 在 locked 状态下，只禁用交互元素而非整个卡片：

```css
.wizard-step-card--locked {
  opacity: 0.55;
}
.wizard-step-card--locked .wizard-step-card__body,
.wizard-step-card--locked .wizard-step-card__footer {
  pointer-events: none;
}
/* 保持 header 可点击（用于展开/折叠） */
```

或更好的方案：不使用 `pointer-events: none`，而是通过 Vue 的 `:disabled` 属性逐个禁用按钮和输入框。

### 3.3 🟠 高优先级：保存配置功能未触发成功反馈

**现象**: 点击 "保存配置" 按钮后，没有出现成功提示消息，配置也未出现在首页列表中。

**可能原因**:
1. 上述 locked 状态阻塞导致按钮点击实际未生效
2. `saveConfigHandler` 中构建的 state 对象格式与 `stateToSnakeCase` 期望的格式不匹配

**代码分析**: [ExportActions.vue:77-110](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step4/ExportActions.vue#L77)

`saveConfigHandler` 手动构建 state 对象，然后传给 `saveConfig`，而 `saveConfig` 内部又调用 `stateToSnakeCase`。但手动构建的 state 对象的 `output` 字段使用 camelCase（如 `outputDir`），`stateToSnakeCase` 期望的也是 camelCase，所以这部分应该没问题。

**主要阻塞原因**: 3.1 的 locked 状态阻塞问题导致 "保存配置" 按钮在 Step 5 locked 时无法被物理点击。

### 3.4 🟡 中优先级：输出配置 sourceTable 下拉选项为空

**现象**: Step 4 输出配置的 "数据源表（source_table）" 下拉选项为空，因为 `sourceTableOptions` 计算属性依赖 `store.processors` 的 `outputTables`，但在初始状态下处理器没有填写输出表名。

**当前代码**: [OutputConfigTab.vue](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step3/OutputConfigTab.vue)

```typescript
const sourceTableOptions = computed(() => {
  const tables: Array<{ label: string; value: string }> = []
  for (const proc of store.processors) {
    for (const t of proc.outputTables) {
      if (t) tables.push({ label: t, value: t })
    }
  }
  return tables
})
```

**优化建议**: 同时包含输入源的表名作为备选：

```typescript
const sourceTableOptions = computed(() => {
  const tables: Array<{ label: string; value: string }> = []
  const seen = new Set<string>()
  // 优先显示处理器输出表
  for (const proc of store.processors) {
    for (const t of proc.outputTables) {
      if (t && !seen.has(t)) { seen.add(t); tables.push({ label: `${t} (处理输出)`, value: t }) }
    }
  }
  // 也包含输入源表名
  for (const inp of store.inputs) {
    if (inp.table && !seen.has(inp.table)) { seen.add(inp.table); tables.push({ label: `${inp.table} (输入源)`, value: inp.table }) }
  }
  return tables
})
```

### 3.5 🟡 中优先级：AI 聊天面板只更新第一个处理器

**现象**: AI 聊天面板的 `onAiSend` 函数在收到 SQL 生成结果时，只更新 `store.processors[0]`，不支持指定更新哪个处理器。

**当前代码**: [ConfigWizardView.vue:355-365](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/views/ConfigWizardView.vue#L355)

```typescript
if (parsed.sql) {
  if (store.processors.length > 0) {
    const proc = { ...store.processors[0], sql: parsed.sql }
    // 始终更新第一个处理器
    store.updateProcessor(0, proc)
  }
}
```

**优化建议**: 添加上下文参数，让 AI 知道当前正在编辑哪个处理器步骤：

```typescript
// 在 context 中添加当前展开的处理器索引
context.currentProcessorIndex = expandedIndex.value

// 在结果处理时使用该索引
const targetIndex = context.currentProcessorIndex ?? 0
if (store.processors.length > targetIndex) {
  const proc = { ...store.processors[targetIndex], sql: parsed.sql }
  store.updateProcessor(targetIndex, proc)
}
```

### 3.6 🔵 低优先级：ProcessorCard 输出表名使用 NInput 而非原生 input

**现象**: ProcessorCard 中的输出表名输入框使用 Naive UI 的 `NInput` 组件，Playwright 无法通过 `input[placeholder="输出表名"]` 定位到它，因为 NInput 渲染为 `<div>` 包裹的 `<input>`，且 placeholder 可能不直接在 input 元素上。

**影响**: 自动化测试定位困难，但不影响用户实际操作。

### 3.7 🔵 低优先级：步骤名称自动编号不够智能

**现象**: 当添加第二个处理器时，默认显示 "步骤 2"，但用户可能期望更语义化的名称（如 "数据过滤"、"数据聚合" 等）。

**优化建议**: 可根据 SQL 内容自动推断步骤名称，或提供名称模板。
