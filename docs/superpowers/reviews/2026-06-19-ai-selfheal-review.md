## 四、逐任务评估

### O-01 弹窗优先显示 AI 诊断 ⭐

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 低风险，DiagnosisPanel 已实现，只需调整显示优先级 |
| 工作量 | ✅ "小"估算合理（修改 2 个文件） |
| 建议 | 无 |

### O-02 autofix 自动填入表单 ⭐

| 维度 | 评估 |
|------|------|
| 可行性 | ⚠️ 中等风险。`applyAutofixes` 需要精确的字段路径映射（step + field → store path） |
| 工作量 | ✅ "大"估算合理（涉 5 个文件，核心逻辑复杂） |
| 风险 | autofix 返回的 field 值可能不匹配实际 store 字段名（如 AI 返回 `sql`，但 store 中是 `processor.sql`） |
| 建议 | 增加字段名映射表，明确 AI 返回字段名与 store 路径的对应关系。例如：`{ "sql": "processors[0].sql", "path": "inputs[0].config.path" }` |

### O-03 修复建议去技术化 ⭐

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 低风险，prompt 优化 |
| 工作量 | ✅ "中"估算合理 |
| 建议 | prompt 中的术语映射表需要定期维护。建议抽取为独立的 `TERMINOLOGY_MAP` 常量方便更新 |

### O-04 "让 AI 帮你重写"入口 ⭐

| 维度 | 评估 |
|------|------|
| 可行性 | ⚠️ 中等风险。orchestrate 的响应质量取决于 AI 模型能力。如果 AI 生成的 SQL 仍然有错，用户体验会更差 |
| 工作量 | ✅ "中"估算合理 |
| 风险 | AI 重写生成的配置可能引入新错误，需要先 precheck 再应用 |
| 建议 | 重写后自动触发 precheck，如果仍有 error 级别问题则提示用户"AI 重写未完全解决，建议手动修复" |

### O-05 诊断增加影响说明 ⭐

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 低风险 |
| 工作量 | ✅ "小"估算合理 |
| 建议 | 无 |

### O-06 异常检测可视化

| 维度 | 评估 |
|------|------|
| 可行性 | ⚠️ "纯 CSS 柱状图"方案可能不够灵活。如果数据列很多（如 20+ 列），柱状图会密集难以阅读 |
| 工作量 | ⚠️ "大"估算可能偏大。纯 CSS 柱状图可以用一个 50 行的组件实现 |
| 建议 | 限制显示列数（如最多 10 列），超出时只显示异常列。工作量可降为"中" |

### O-07 定时任务自动诊断

| 维度 | 评估 |
|------|------|
| 可行性 | ⚠️ 中风险。`auto_diagnose` 是 async 函数，在同步 scheduler 中需要用 `asyncio.run()` 调用。如果 scheduler 已有 running loop 会冲突 |
| 工作量 | ✅ "中"估算合理 |
| 建议 | 先检查 scheduler 是否已经有 event loop。如果没有，用 `asyncio.new_event_loop()` + `loop.run_until_complete()` 会更安全 |

### O-08 诊断缓存去重

| 维度 | 评估 |
|------|------|
| 可行性 | ✅ 低风险 |
| 工作量 | ✅ "中"估算合理 |
| 建议 | 缓存 key 建议改为 `config_id + step + error_message_hash`，因为同一配置不同步骤可能报不同错误 |

### O-09 多轮对话支持

| 维度 | 评估 |
|------|------|
| 可行性 | ⚠️ 高风险。AiChatPanel 在 GuidePanel 重构后已不再集成到主流程中，重新集成需要处理两个 AI 交互入口的冲突 |
| 工作量 | ✅ "大"估算合理 |
| 建议 | 考虑更轻量的方案：在 DiagnosisPanel 底部增加一个输入框 + "追问"按钮，直接调 chat API，不需要完整集成 AiChatPanel |

### O-10 诊断趋势分析
