# ConfigForge AI 引导式配置方案 — 设计评审文档

> 评审日期：2026-06-07
> 评审对象：`docs/superpowers/specs/2026-06-07-ai-guided-configuration-design.md` v1.0
> 评审结论：**方向正确，需修正 5 个核心问题后方可进入开发**

---

## 一、总体评价

| 维度 | 评价 | 说明 |
|------|------|------|
| 产品方向 | ✅ 正确 | 从"手动填表 + AI 可选辅助"转向"AI 引导 + 用户审核"是正确的演进方向 |
| 流程设计 | ✅ 合理 | 5 步渐进式引导符合用户心智模型 |
| 交互设计 | ⚠️ 部分过于复杂 | AI 面板跟随步骤卡片、自动避让等特性实现成本高，收益低 |
| 降级策略 | ❌ 不完整 | AI 不可用时首页体验断崖，缺少双入口设计 |
| API 设计 | ⚠️ 缺少细节 | 新增 API 与现有 API 的关系不清晰 |
| 技术可行性 | ⚠️ 有风险 | 跨步骤联动、字段追踪等设计实现复杂度高 |

## 二、核心问题（必须修正）

### 问题 1：AI 不可用时首页体验断崖

**方案原文（3.1 节 + 6.2 节）**：
- 首页只有一个"描述你的需求"输入框
- AI 不可用时"面板不出现，向导正常使用（降级为纯手动模式）"

**问题**：
- 首页唯一入口是 AI 输入框，AI 不可用时这个输入框变成什么？灰掉？隐藏？
- "试试这样说"的示例 chips 还显示吗？显示的话用户点了没反应，不显示的话首页空荡荡
- 用户输入了需求描述后点"开始配置"，但 AI 不可用，描述去哪了？丢失了？

**建议**：首页设计**双入口路径**：

```
┌──────────────────────────────────────────────┐
│  ⚡ ConfigForge                              │
├──────────────────────────────────────────────┤
│                                              │
│         ⚡ AI 驱动 · 你说需求，AI 填配置        │
│                                              │
│  ┌──────────────────────────────────────┐    │
│  │ 💬 描述你的数据处理需求...          🎤 │    │  ← AI 可用时高亮
│  └──────────────────────────────────────┘    │
│              [✨ 开始配置]                     │  ← AI 可用时主入口
│                                              │
│         或 手动创建 →                         │  ← 始终可见的备选入口
│                                              │
│  试试这样说：                                  │  ← AI 可用时才显示
│  · 导出用户表到 CSV                           │
│  · 合并订单表和用户表，统计金额                  │
│                                              │
│  ──── 最近配置 ────                           │
│  ...                                         │
└──────────────────────────────────────────────┘
```

**降级行为**：
- AI 不可用：输入框灰掉 + "需先配置 AI 才能使用" 提示 + "前往设置" 按钮
- "手动创建 →" 链接始终可见，点击直接进入向导（无 AI 面板）
- 示例 chips 在 AI 不可用时隐藏

### 问题 2：AI 面板定位策略过于复杂

**方案原文（8.1 节）**：
- 默认跟随当前步骤卡片右侧
- 可拖拽到任意位置
- 自动避让"下一步"按钮等控件
- 固定定位，不跟随滚动

**问题**：4 个特性组合的实现复杂度极高：

| 特性 | 实现难点 |
|------|---------|
| 跟随步骤卡片 | 需监听每个步骤卡片 DOM 位置，步骤切换时面板动画移动，需处理卡片高度不同导致的对齐问题 |
| 可拖拽 | 需实现 drag & drop，拖拽后是否还跟随？拖拽到步骤卡片上方时怎么处理？ |
| 自动避让 | 需碰撞检测，判断面板是否遮盖"下一步"按钮，遮盖时面板往哪移？ |
| 固定定位 + 不跟随滚动 | 用户滚动时面板不动但步骤卡片在动，位置关系持续变化，跟随逻辑失效 |

**建议**：v1.0 简化为**固定右侧面板**：

```
┌──────────────────────────────┬────────────────────┐
│  步骤内容区                    │  🤖 Forge · AI 助手  │
│  (flex-1, 可滚动)             │  (固定宽度 360px)    │
│                              │                    │
│  Step 1: 场景信息             │  AI 消息区域         │
│  Step 2: 输入源               │  (可滚动)           │
│  Step 3: 处理步骤             │                    │
│  ...                         │  [输入框] [发送]     │
└──────────────────────────────┴────────────────────┘
```

- 复用现有 `AiChatPanel.vue` 的 sidebar 模式
- 面板始终在右侧，不跟随步骤，不拖拽
- 最小化时缩成右侧悬浮按钮（保留 8.2 节设计）
- v2.0 再考虑拖拽和跟随

### 问题 3：跨步骤联动检测设计不够务实

**方案原文（4.4 节）**：
- 用户修改 SQL 的输出列 → AI 检测到变化 → 主动提示是否需要同步更新 Step 4 的列映射
- 气泡样式：⚠ 警告色（黄色边框），需要用户确认

**问题**：

1. **谁来做检测？** 前端 watch SQL 变化然后调 AI？成本太高。前端自己做 diff？需要解析 SQL 的 SELECT 列。
2. **时机问题**：用户正在编辑 SQL，每敲一个字都触发检测？还是失焦时？失焦检测也有问题——用户可能只是暂时切走。
3. **误报**：SQL 改了注释、改了 WHERE 条件但没改 SELECT 列，不应该触发。
4. **AI 调用成本**：每次检测都调 AI 意味着大量 API 请求和延迟。

**建议**：v1.0 改为**步骤切换时检测**：

```
用户在 Step 3 点"下一步" → 进入 Step 4 时
  → 前端比较 SQL SELECT 列 与 Step 4 已有列映射
  → 如果不同，弹轻量提示条：
     "SQL 输出列已变更（新增 2 列，移除 1 列），是否更新列映射？"
     [更新列映射] [保持不变]
```

- 检测逻辑纯前端，不调 AI
- 只在步骤切换时触发，不打断编辑
- 误报率低（只比较列名，不分析 SQL 语义）

### 问题 4：新增 API 与现有 API 的关系不清晰

**方案原文（7.2 节）**：
- `POST /api/ai/guide/start` — 新增
- `POST /api/ai/guide/step` — 新增
- `POST /api/ai/generate-code` — 已有
- `POST /api/ai/mapping` — 已有

**问题**：

1. 现有 `POST /api/ai/suggest` 已支持 `category: scene|columns|sql|mapping|chat`，新 API 是替代还是并行？
2. `guide/start` 返回什么？场景名称 + 描述 + 建议输入类型？还是完整 5 步配置？返回太多用户没审核机会，太少引导感不强。
3. `guide/step` 的"预填数据"格式是什么？直接写入 store 的 JSON？需前端解析的中间格式？
4. 两套并行 API 会导致后端维护成本翻倍。

**建议**：

**复用现有 `suggest` API，不新增独立 API**：

```
guide/start → 调用 suggest(category='scene', context=用户需求描述)
  返回：{scene_name, description, suggested_input_type}

guide/step → 调用 suggest(category=当前步骤对应category, context=步骤上下文)
  返回：{message, prefill?}
```

- `suggest` API 已有速率限制、错误清洗、超时处理等基础设施
- 只需在 `suggest` 的 context 中增加 `current_step` 和 `wizard_state` 字段
- 前端新增一个 `useAiGuide` composable 封装引导逻辑，内部调 `suggest`
- 后端只需增强 `suggest` 的 prompt 模板，不需要新端点

**`guide/step` 返回格式建议**：

```typescript
interface GuideResponse {
  message: string           // AI 引导消息（显示在面板中）
  actions?: GuideAction[]   // 快捷操作按钮
  prefill?: {               // 可选：AI 建议预填的数据
    [fieldPath: string]: any  // 如 "scene.name": "订单统计报表"
  }
}

interface GuideAction {
  label: string             // 按钮文字
  value: string             // 点击后发送的消息
  style?: 'primary' | 'default' | 'warning'
}
```

### 问题 5：字段来源追踪设计过于复杂

**方案原文（7.3 节）**：
- `aiGeneratedFields: Set<string>` 追踪哪些字段是 AI 填写的

**问题**：

1. **路径表示**：`scene.name`？`processors[0].sql`？`output.config.columns[2].source`？嵌套数组索引不稳定。
2. **状态转换**：用户手动修改后，从 Set 中移除？还是标记为"已编辑"？移除后丢失了"AI 曾经填过"的信息。
3. **AI 对话修改**：用户说"把统计改成按产品类型分组"，AI 修改了 SQL。这算 AI 填写还是用户修改？
4. **编排模式**：`orchestrate` 一次填入多个处理器，每个处理器的每个字段都要标记？
5. **持久化**：这个 Set 要不要存 localStorage？刷新后还保留 AI 标记吗？

**建议**：**简化为 UI 层临时标记，不持久化**：

```typescript
// wizard.ts
const aiPrefilledFields = ref<Map<string, boolean>>(new Map())

// AI 填写时标记
function markAiPrefilled(fieldPath: string) {
  aiPrefilledFields.value.set(fieldPath, true)
}

// 用户手动编辑时移除标记（通过 watch 或 @input 事件）
function markUserEdited(fieldPath: string) {
  aiPrefilledFields.value.delete(fieldPath)
}

// 不持久化，刷新后标记消失
```

- 标记仅用于 UI 视觉区分（绿色边框 + "AI 已填写" badge）
- 刷新页面后标记消失，所有字段一视同仁
- 避免了复杂的状态管理和持久化问题

## 三、设计决策评估

| 决策 | 评估 | 建议 |
|------|------|------|
| 8.1 面板跟随步骤卡片 | ⚠️ 过于复杂 | v1.0 改为固定右侧面板，复用 AiChatPanel sidebar 模式 |
| 8.2 最小化缩成悬浮图标 | ✅ 合理 | 建议单击恢复（而非双击），符合用户习惯 |
| 8.3 语音输入占位符 | ❌ 不建议 | 占位符让用户期待功能存在，点击后"即将支持"令人失望。建议 AI 可用时才显示 🎤，不可用时直接不显示 |
| 8.4 对话历史保留 | ✅ 合理 | localStorage 即可，注意 5MB 限制，长对话需截断（保留最近 50 条） |
| 8.5 仅中文 | ✅ 合理 | 先做好中文，后续 i18n |
| 8.6 移动端 bottom sheet | ✅ 合理 | v1.0 就应考虑，现有 AiChatPanel 已有 fullscreen 模式可直接复用 |

## 四、方案缺失的关键点

### 4.1 AI 响应延迟处理

AI 生成代码可能需要 5-15 秒，方案未说明等待期间面板显示什么。

**建议**：
- 面板显示打字指示器（三个点动画）+ "AI 正在思考..."
- 超过 5 秒追加提示 "生成较复杂的代码可能需要更长时间"
- 超过 30 秒显示"取消"按钮
- 现有 `AiChatPanel` 已有打字机效果和 loading 状态，可复用

### 4.2 AI 生成内容有误的回退

方案只说了"生成失败"场景，没说"生成了但有问题"场景。

**建议**：
- SQL 语法错误：后端 dry_run 失败时，面板自动显示错误信息 + "重新生成"按钮
- 逻辑错误：用户说"不对"，AI 基于对话历史修正
- 部分正确：用户可直接编辑代码，AI 不干预（用户编辑后移除 AI 标记）

### 4.3 与现有 AiChatPanel 的关系

方案新增 `AiGuidePanel.vue`，但现有 `AiChatPanel.vue` 已有完整聊天功能。

**建议**：
- **不新增组件**，在现有 `AiChatPanel.vue` 基础上增强
- 增加引导模式（guide mode）：AI 主动发消息 + 快捷操作按钮
- 增加步骤感知：面板知道当前步骤，消息带步骤标记
- 保留自由聊天模式作为备选（用户可以随时切换到自由对话）

### 4.4 AI 消息的步骤标记格式

方案说"每条消息带有步骤标记"，但未定义格式。

**建议**：

```typescript
interface ChatMessage {
  role: 'user' | 'ai'
  content: string
  step?: number              // 关联的步骤编号（1-5）
  type?: 'guide' | 'chat' | 'warning' | 'suggestion'  // 消息类型
  actions?: GuideAction[]    // 快捷操作按钮
  prefill?: Record<string, any>  // AI 建议预填的数据
  timestamp: number
}
```

### 4.5 "确认，下一步"按钮的行为

AI 面板中的"确认"按钮是只确认当前步骤，还是同时触发"下一步"导航？

**建议**：只确认，不自动导航。让用户自己点"下一步"，保持控制感。AI 不应该替用户做导航决策。

## 五、现有代码基础与方案的差距分析

| 现有能力 | 方案要求 | 差距 | 建议 |
|---------|---------|------|------|
| AiChatPanel sidebar/overlay/fullscreen 三模式 | 固定右侧面板 + 最小化 | 小 | 增强现有组件，不新建 |
| `suggest` API 6 个 category | guide/start + guide/step | 中 | 复用 suggest，增强 context |
| `orchestrate` 多步骤编排 | AI 逐步引导 | 大 | 编排作为"高级模式"保留，引导作为"标准模式"新增 |
| `useAiApi` composable | `useAiGuide` composable | 中 | 新增引导 composable，内部调 useAiApi |
| AiInlineTip 5 处步骤提示 | AI 面板主动引导 | 大 | AiInlineTip 保留作为轻量提示，AI 面板做深度引导 |
| AiStatusBanner 检测 AI 可用性 | 首页双入口 | 中 | 复用 useAiStatus 控制首页入口显示 |
| `translate-checkpoint` 后端已实现前端未接入 | 未提及 | — | 建议在 Step 3 引导中接入此能力 |
