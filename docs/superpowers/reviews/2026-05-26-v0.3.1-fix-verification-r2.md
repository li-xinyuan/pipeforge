### P1-3：OrchestrationStep 类型未使用 discriminated union

| 检查项 | 结果 | 说明 |
|--------|------|------|
| `OrchestrationStep` 改为 discriminated union | ✅ | `wizard.ts` 中已改为 `plugin: 'sql'` + `sql: string` / `plugin: 'python'` + `script: string` |
| `ProcessorStep` discriminated union | ✅ | 同步更新 |
| `SnakeState.processors` discriminated union | ✅ | `serialization.ts` 同步更新 |
| **OrchestrationResult.vue 类型安全** | ❌ | `step.script \|\| step.sql` 在 discriminated union 上不合法 |
| **askOrchestrate 返回类型** | ❌ | 仍为旧类型，缺少 `plugin` 和 `script` 字段 |
| **vue-tsc 类型检查** | ❌ | **3 个 TypeScript 错误** |

**问题详情**：

1. **`OrchestrationResult.vue:12`** — `step.script || step.sql` 在 discriminated union 上不合法：
   - `step.script` 不存在于 SQL 变体 `{ plugin: 'sql'; sql: string }`
   - `step.sql` 不存在于 Python 变体 `{ plugin: 'python'; script: string }`

2. **`ConfigWizardView.vue:552`** — `askOrchestrate` 返回类型与 `OrchestrationResult` 不兼容：
   - 返回类型声明为 `{ name: string; input_tables: string[]; output_tables: string[]; sql: string }[]`
   - 缺少 `plugin` 和 `script` 字段
   - 导致 `aiMessages.value.push({ role: 'ai', content: '', orchestration: result })` 类型错误

3. **`askOrchestrate` 返回类型**（[useWizardApi.ts:114-118](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/composables/useWizardApi.ts#L114-L118)）：
   ```typescript
   steps: Array<{ name: string; input_tables: string[]; output_tables: string[]; sql: string }>
   ```
   应改为与 `OrchestrationStep` 一致的 discriminated union。

**修复建议**：

1. `OrchestrationResult.vue:12` 改为条件渲染：
   ```vue
   <code>{{ step.plugin === 'python' ? step.script : step.sql }}</code>
   ```

2. `useWizardApi.ts` 的 `askOrchestrate` 返回类型改为：
   ```typescript
   steps: Array<
     | { name: string; plugin: 'sql'; input_tables: string[]; output_tables: string[]; sql: string }
     | { name: string; plugin: 'python'; input_tables: string[]; output_tables: string[]; script: string }
   >
   ```

3. `ConfigWizardView.vue:552` 的类型断言或转换需要同步调整

## 三、新发现的问题

### 🔴 新问题 1：WizardStepCard 锁定遮罩定位失效

**级别**：P0（阻断性 Bug）
**位置**：[WizardStepCard.vue:50](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/wizard/WizardStepCard.vue#L50)
**现象**：锁定步骤的遮罩层 `position: absolute; inset: 0` 无法正确定位到卡片上，因为父元素 `.wizard-step-card` 缺少 `position: relative`。遮罩会相对于最近的 positioned 祖先定位，导致遮罩覆盖范围错误。
**修复**：在 `.wizard-step-card` CSS 中添加 `position: relative;`

### 🔴 新问题 2：OrchestrationStep discriminated union 导致 3 个 TypeScript 编译错误

**级别**：P0（阻断性 Bug — TypeScript 编译失败）
**位置**：
- [OrchestrationResult.vue:12](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/wizard/OrchestrationResult.vue#L12) — `step.script || step.sql` 在 union 上不合法
- [ConfigWizardView.vue:552](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/views/ConfigWizardView.vue#L552) — `askOrchestrate` 返回类型与 `OrchestrationResult` 不兼容
- [useWizardApi.ts:114](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/composables/useWizardApi.ts#L114) — 返回类型缺少 `plugin` 和 `script` 字段

**现象**：`vue-tsc --noEmit` 报 3 个错误，CI 构建会失败。

**修复**：
1. `OrchestrationResult.vue:12` → `{{ step.plugin === 'python' ? step.script : step.sql }}`
2. `useWizardApi.ts` 的 `askOrchestrate` 返回类型改为与 `OrchestrationStep` 一致的 discriminated union
3. `ConfigWizardView.vue:552` 可能需要类型断言 `as OrchestrationResult`

### 🟡 新问题 3：Python "API 调用"模板依赖未安装的 `requests` 库

**级别**：P1
**位置**：[PythonProcessorContent.vue:118](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step3/PythonProcessorContent.vue#L118)
**现象**：API 调用模板包含 `import requests`，但当前 Python 环境未安装 `requests`。用户点击模板后执行 dry-run 会报 `ModuleNotFoundError`。
**修复建议**：
- 方案 A：将模板改为使用标准库 `urllib.request`
- 方案 B：在模板标签旁添加提示"需要安装 requests 库"
- 方案 C：在 dry-run 前检测依赖并提示安装

### 🟢 新问题 4：Python 模板中硬编码的表名 `source`

**级别**：P2
**位置**：[PythonProcessorContent.vue:115-118](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step3/PythonProcessorContent.vue#L115-L118)
**现象**：所有 4 个模板都硬编码了 `FROM source` 或 `FROM source WHERE`，但用户的输入表名可能不是 `source`。SQL 步骤有自动填充 `FROM "实际表名"` 的逻辑，但 Python 模板没有。
**修复建议**：模板填入时，将 `source` 替换为当前可用的第一个输入表名（类似 SQL 步骤的 `fromTable` 逻辑）。

### 🟢 新问题 5：PythonProcessorContent 的 `proc` prop 类型为 `ProcessorStep` 而非 `Extract<ProcessorStep, { plugin: 'python' }>`

**级别**：P2（类型安全）
**位置**：[PythonProcessorContent.vue:91](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step3/PythonProcessorContent.vue#L91)
**现象**：`props.proc` 类型声明为 `ProcessorStep`（联合类型），而非精确的 Python 变体。虽然 `pyProc` computed 做了运行时检查，但 props 类型不够严格，允许传入 SQL 步骤。
**对比**：`SqlProcessorContent.vue` 已使用 `Extract<ProcessorStep, { plugin: 'sql' }>` 作为 props 类型。
**修复建议**：将 `proc: ProcessorStep` 改为 `proc: Extract<ProcessorStep, { plugin: 'python' }>`，与 SQL 组件保持一致。
