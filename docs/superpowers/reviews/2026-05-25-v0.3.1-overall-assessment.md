### 3.2 关键 UX 发现

#### 🔴 P0：5 个步骤卡片同时显示

**现象**：页面加载后，5 个 `wizard-step-card` 同时可见，其中 4 个被标记为 `--locked`（`opacity: 0.55` + `pointer-events: none`）。

**影响**：用户首次进入页面时看到大量灰色锁定内容，造成信息过载和认知负担。锁定步骤虽然不可交互，但内容仍然可见（包括按钮、输入框等），用户可能困惑是否可以操作。

**建议**：
- 方案 A：只渲染当前步骤和已完成的步骤，未解锁步骤仅显示标题
- 方案 B：锁定步骤折叠为仅标题行，点击展开时提示"请先完成上一步"
- 方案 C（最小改动）：锁定步骤的 `opacity` 降至 `0.3`，并添加遮罩层 + 锁图标

#### 🟡 P1：`pulse-cta` 动画导致按钮点击不稳定

**现象**：`pulse-cta` CSS 动画（`box-shadow` 脉冲效果）导致 Playwright 报 "element is not stable" 错误，需要 `force: true` 才能点击。

**影响**：虽然真实用户点击不受影响（浏览器会等待动画帧），但自动化测试困难。更重要的是，`pulse-cta` 动画在 `box-shadow` 上使用 `transition`，可能导致按钮在动画过程中视觉"抖动"。

**建议**：将 `pulse-cta` 动画从 `box-shadow` 改为 `outline` 或 `border-color`，避免触发布局重计算。

#### 🟡 P2：Python 处理器缺少"AI 生成"能力

**现象**：SQL 处理器有"✨ AI 生成 SQL"按钮，但 Python 处理器没有任何 AI 辅助功能。

**影响**：用户需要从零编写 Python 脚本，包括 `def process(ctx):` 函数签名和 `ctx.db.connection` API 调用，门槛较高。

**建议**：v0.3.2 可考虑：
- 添加 Python 脚本模板选择（数据清洗、正则提取、API 调用等）
- 添加"AI 生成 Python"按钮（需后端支持 `category: "python"` prompt）
- 在 textarea 下方添加 `ctx` API 速查卡片

#### 🟢 P3：`onOrchestrateConfirm` 中 Python 步骤的 `script` 回退到 `s.sql`

**位置**：[ConfigWizardView.vue:570](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/views/ConfigWizardView.vue#L570)

```typescript
script: s.script || s.sql || ''
```

**说明**：当 `plugin === 'python'` 但 `s.script` 为空时，会回退到 `s.sql`。这是为了兼容当前 AI 编排只返回 `sql` 字段的情况（P3 延后），但语义上不太准确。

**影响**：低 — 当前 AI 编排不会返回 Python 步骤，此回退逻辑不会被触发。

**建议**：在 AI 编排支持 Python 步骤后移除 `|| s.sql` 回退。

## 五、代码质量问题

### 5.1 🟡 `OrchestrationStep.sql` 仍为必填字段

**位置**：[wizard.ts:30](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/types/wizard.ts#L30)

```typescript
export interface OrchestrationStep {
  name: string
  plugin?: 'sql' | 'python'
  input_tables: string[]
  output_tables: string[]
  sql: string        // ← 必填，但 Python 步骤不需要
  script?: string    // ← 可选，但 Python 步骤必须有
}
```

**问题**：`OrchestrationStep` 的 `sql` 是必填字段，`script` 是可选字段。但 Python 步骤不需要 `sql`，且必须有 `script`。这与 `ProcessorStep` 的 discriminated union 设计不一致。

**建议**：改为 discriminated union：

```typescript
export type OrchestrationStep =
  | { name: string; plugin: 'sql'; input_tables: string[]; output_tables: string[]; sql: string }
  | { name: string; plugin: 'python'; input_tables: string[]; output_tables: string[]; script: string }
```

### 5.2 🟡 `PythonProcessorContent.vue` 的 `dryRun` 复用 SQL dry-run API

**位置**：[PythonProcessorContent.vue:98](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step3/PythonProcessorContent.vue#L98)

```typescript
const result = await runDryRunApi(store.$state)
```

**说明**：Python 处理器的"预览结果"复用了 SQL 的 dry-run API。该 API 会将整个 `WizardState` 发送到后端，后端通过 `build_yaml` → `pipeforge run --dry-run` 执行。这意味着 Python 脚本会在 dry-run 中真正执行（包括 `exec()` 和 `process(ctx)` 调用）。

**潜在风险**：
- dry-run 本应只验证不执行，但 Python 步骤会真正执行
- 如果 Python 脚本有副作用（如网络请求、文件写入），dry-run 阶段就会触发
- 超时控制依赖 `signal.SIGALRM`，在 dry-run 的 SQLite 环境中可能行为不同

**建议**：
- 在 dry-run 模式下对 Python 步骤仅做 AST 校验（检查 `process` 函数存在），不实际执行
- 或者明确在 UI 上标注"预览将执行 Python 脚本"

### 5.3 🟢 `inferPythonStepName` 的规则过于简单

**位置**：[SqlEditorTab.vue:155](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step3/SqlEditorTab.vue#L155)

```typescript
function inferPythonStepName(script: string): string {
  if (/DELETE\s+FROM/i.test(script)) return '数据清洗'
  if (/CREATE\s+TABLE/i.test(script) && /WHERE/i.test(script)) return '数据过滤'
  if (/GROUP\s+BY/i.test(script)) return '数据聚合'
  if (/JOIN/i.test(script)) return '表连接'
  if (/import\s+re|re\.sub|re\.match|re\.findall/i.test(script)) return '正则提取'
  if (/import\s+requests|urllib|httpx/i.test(script)) return 'API 调用'
  return 'Python 处理'
}
```

**问题**：前 4 条规则（DELETE FROM、CREATE TABLE + WHERE、GROUP BY、JOIN）是 SQL 关键字匹配，对 Python 脚本来说，这些匹配的是 Python 代码中的 SQL 字符串内容，而非 Python 代码本身的结构。这可能导致误判。

**建议**：
- Python 步骤名称推断应优先匹配 Python 特征（`import re`、`import requests`）
- SQL 关键字匹配应降级为辅助判断
- 或者直接让用户手动命名，不做自动推断

### 5.4 🟢 `OutputConfigTab.vue` 中"从 SQL 推断列"按钮对 Python 步骤的适配

**位置**：[OutputConfigTab.vue:321](file:///Users/lixinyuan/code/CCTEST/configforge-web/src/components/step3/OutputConfigTab.vue#L321)

```typescript
const sql = p0 ? (p0.plugin === 'sql' ? p0.sql : p0.script) : ''
const cols = inferSelectColumns(sql)
```

**说明**：当第一个处理器是 Python 步骤时，`inferSelectColumns` 会尝试从 Python 脚本中解析 SQL 的 `SELECT` 列名。这对 Python 脚本基本无效，因为 `inferSelectColumns` 是为 SQL 设计的。

**影响**：低 — CSV 输出模式下会自动调用 `onInferColumns`，如果推断失败会 fallback 到执行 SQL 查询，Python 脚本会真正执行并返回列信息。

**建议**：对 Python 步骤跳过 `inferSelectColumns`，直接执行 dry-run 获取列信息。

### 6.2 🟡 Windows 平台无超时保护

**位置**：[python.py:28](file:///Users/lixinyuan/code/CCTEST/src/pipeforge/plugins/processor/python.py#L28)

```python
if sys.platform != "win32":
    import signal
    # ... SIGALRM 超时控制
```

**问题**：Windows 不支持 `signal.SIGALRM`，当前代码在 Windows 上运行 Python 步骤时无超时保护。如果脚本包含无限循环，将永远阻塞。

**建议**：使用 `threading.Timer` + `ctypes` 强制抛出异常作为 Windows 替代方案，或在文档中明确标注 Windows 不支持超时控制。

v0.3.1 的核心功能（AI 编排步骤链 + Python 处理器）已完整实现，所有自动化测试通过（前端 113 + 后端 263），TypeScript 类型检查无错误，原有功能无回归。
