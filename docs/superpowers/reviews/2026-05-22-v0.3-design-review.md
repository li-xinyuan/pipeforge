## 一、方向判定：✅ 正确

设计文档三大能力与 `NEXT_STEPS.md` Phase 2 高度一致：

| 设计文档 | NEXT_STEPS.md Phase 2 | 一致性 |
|---------|----------------------|--------|
| A — 多步 Pipeline | 🔴 多步 Pipeline（链式执行） | ✅ 完全一致 |
| G — AI 自动编排 | 🔴 AI 深度集成 → 自然语言查询 | ✅ 一致，编排是更务实的切入点 |
| B — 错误体验优化 | 🔴 错误体验优化 | ✅ 完全一致 |

**方向正确性论证：**

1. **多步 Pipeline 是当前最大功能天花板** — 当前 `ProcessorConfig` 只有 `plugin: 'sql'` + 单条 SQL + 单个 `outputTable`，任何真实业务场景都需要多步处理（清洗→聚合→格式化），用户只能写超长嵌套 SQL
2. **AI 编排是 ConfigForge 核心差异化** — NEXT_STEPS.md 明确指出"让不懂代码的人也能创建可重复的数据处理流程"，AI 编排直接降低从零到一的成本
3. **错误体验是业务用户的基本门槛** — 当前前端错误处理只有 `useErrorHandler.ts` 的简单 message 显示（16 行代码），API 错误仍是英文原始异常

## 二、必要性判定

| 能力 | 必要性 | 理由 |
|------|--------|------|
| A — 多步 Pipeline | **强必要** | 单 SQL → 单表是硬性天花板。`SqlEditorTab.vue` 已预留了 Jinja2/Python 的 v0.3/v0.4 标签位，说明团队早有规划 |
| B — 错误体验优化 | **必要** | 业务用户看到 `sqlite3.OperationalError: no such column: dept_name` 完全无法自助修复。v0.2.1 的 M1 修复做了后端错误脱敏，但前端展示和中文提示还没做 |
| G — AI 自动编排 | **有条件必要** | AI 编排依赖多步 Pipeline 先落地。没有多步 Pipeline，AI 编排只能生成单条 SQL，价值有限 |

## 三、急需度判定：A > B > G

| 能力 | 急需度 | 分析 |
|------|--------|------|
| A — 多步 Pipeline | **急需** | 这是 v0.2.1 的最大功能缺陷。当前用户需要"先清洗再聚合"只能写一条超长嵌套 SQL |
| B — 错误体验 | **较急** | 不急但不做会持续影响用户信心。后端已做错误脱敏，前端中文提示是自然延伸 |
| G — AI 编排 | **不急** | AI 编排是锦上添花。当前 AI chat 模式已能生成单步 SQL，编排是多步 Pipeline 的自然延伸 |

**分期交付方案（设计文档已采纳）：**

| 版本 | 内容 | 理由 |
|------|------|------|
| **v0.3.0** | A（多步 SQL Pipeline）+ B（错误体验第一层） | 核心能力扩展 + 基础体验保障 |
| **v0.3.1** | G（AI 编排，仅 SQL 步骤）+ B（错误体验第二层） | 依赖 A 的 Processor 链模型稳定后再做 |
| **v0.4.0** | Python Processor + 沙箱方案 + AI Python 编排 | 需要沙箱方案，安全风险大 |

### 4.1 A — 多步 Pipeline 可行性

**有利因素：**

1. `engine.py` 已遍历所有 `processors`（`for proc_spec in self.config.processors`），不是 `processors[:1]`，引擎基础已就绪
2. `ProcessorSpec` 已有 `output_tables: list[str]`，只需增加 `input_tables`
3. SQLite 天然支持多表共存，中间表不需要额外持久化

**风险点：**

1. **DAG 拓扑排序**：需要正确处理循环依赖检测，但算法成熟（Kahn's algorithm），实现简单 — 🟢 低风险
2. **前端 Step 3 改造量大**：当前 `SqlEditorTab.vue` 是单 SQL 编辑器，改为 Processor 链式列表需要重构整个 Step 3 组件结构 — 🟡 中风险
3. **拖拽排序**：已明确推迟到 v0.3.x — � 风险已消除

### 4.2 G — AI 编排可行性

**初评 4 个风险，更新后设计文档已全部回应：**

| 风险 | 初评 | 复评 | 设计文档回应 |
|------|------|------|------------|
| G1: JSON 可靠性 | 🔴 高 | 🟡 中 | ✅ 只生成 SQL 步骤，不生成 Python；结构简化 |
| G2: Python 代码安全 | 🔴 高 | 🟢 已消除 | ✅ Python Processor 推迟到 v0.4.0，v0.3.1 不涉及 |
| G3: 上下文窗口 | 🟡 中 | 🟢 低 | ✅ 只传表名+列名，不传样本数据 |
| G4: 审核交互复杂 | 🟡 中 | 🟢 低 | ✅ MVP 只做"确认全部/重新生成"，不做单步修改 |

**新增降级策略（第五章 5.3 节）** — 设计文档补充了 AI 返回格式错误时的 4 级降级：
1. JSON 解析失败 → 展示原始文本，引导重试
2. 缺少必要字段 → 跳过不合规步骤
3. 完全无法解析 → 降级为单步 SQL 生成
4. 连续 3 次失败 → 建议切换手动模式

**复评结论：G 的风险从 🔴 高降至 🟡 中，可接受。**

### 4.3 B — 错误体验可行性

**完全可行，纯工程量：**
- 后端已有 `ErrorResponse` 模型和错误码体系
- 前端已有 `useErrorHandler.ts` 基础设施
- 只需增加错误码→中文消息的映射表

### 7.2 残留关注点

以下不是设计文档的问题，而是实施过程中需要注意的事项：

| # | 关注点 | 说明 | 建议 |
|---|--------|------|------|
| 1 | `execute_dry_run` 实现细节 | 设计文档只描述了"执行 input + processor，跳过 output"，但多步 Pipeline 的 dry_run 是只执行当前步骤还是执行到当前步骤？ | 建议实现为"执行到当前步骤"，即按 DAG 顺序执行该步骤及其所有前置步骤，这样用户可以看到完整的中间数据 |
| 2 | DAG 并行执行 | 设计文档 3.3 节提到 `asyncio.gather` 并行执行无依赖步骤，但 SQLite 不支持真正的并发写入 | 建议并行只用于 SQL 编译/验证阶段，实际执行仍串行；或使用 WAL 模式 + 连接池 |
| 3 | 中间表清理策略 | 设计文档提到"错误时不清理 SQLite"，但成功执行后中间表是否保留？如果保留，多次执行会累积 | 建议成功执行后保留最终输出表，清理中间表（`_tmp_` 前缀）；或提供 `--keep-intermediate` 选项 |
| 4 | `input_tables` 为空的语义 | 第一个 processor 的 `input_tables` 应该来自 input plugins 产出的表，设计文档未明确 | 建议引擎在拓扑排序前，将 input plugins 产出的表名注入 `available` 集合，第一个 processor 的 `input_tables` 必须是这些表名的子集 |
| 5 | AI 编排 prompt 的 `orchestrate` 类别 | 需要在 `orchestrator.py` 的 `SYSTEM_PROMPTS` 中新增，与现有 `scene`/`columns`/`sql` 等并列 | v0.3.1 实施时注意 prompt 工程迭代，建议准备 5-10 个测试用例验证 JSON 输出格式 |
