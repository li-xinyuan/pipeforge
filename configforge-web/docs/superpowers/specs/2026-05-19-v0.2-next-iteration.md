## 背景

PipeForge 引擎 v0.1 MVP 已完全实现，且 CSV 输入/输出和 SQL Jinja2 变量替换已提前完成。ConfigForge 前端重设计核心架构已到位（单页滚动、AI 面板、暗色模式、响应式），但部分视觉细节和 AI 交互覆盖不足。

本次迭代目标：前端补齐设计 spec 中最显眼的两个缺口 + 引擎解锁 v0.2 最关键的功能。

## 改动清单

### 前端

#### F1: 每步独立色彩标识

- **现状**: WizardStepCard.vue 中 5 个步骤的图标区使用统一青瓷绿底色，且当前无区分步骤的 prop
- **目标**: 每个步骤卡片图标区显示不同底色
- **文件**: `src/components/WizardStepCard.vue`、`src/views/ConfigWizardView.vue`
- **方案**:
  - WizardStepCard 新增 `iconColor?: string` prop，模板中绑定为 `background` 样式
  - ConfigWizardView 中每个 `<WizardStepCard>` 传入对应色值
- **色值（浅色模式）**:

| 步骤 | 底色 | 图标色 |
|------|------|--------|
| Step 1 | `#f0fdfa` | `#0d9488` |
| Step 2 | `#ccfbf1` | `#0f766e` |
| Step 3 | `#fef3c7` | `#d97706` |
| Step 4 | `#fef2f2` | `#dc2626` |
| Step 5 | `#eff6ff` | `#0284c7` |

- **暗色模式**: 通过 CSS 变量在 `[data-theme="dark"]` 下覆写各步骤底色为深色对应版本（`#0a2a1e`、`#0a2e1e`、`#2d2006`、`#2d0a0a`、`#0a1a2e`），图标色保持不变

#### F2: AI 内联建议 Step2-5 覆盖

- **现状**: AiInlineTip.vue 仅在 Step1 展示
- **目标**: 每个步骤卡片内都有条件展示的 AI 提示
- **文件**: `src/components/AiInlineTip.vue`（小改）、`src/views/ConfigWizardView.vue`（每步加条件块）
- **触发条件（精确表达式）**:

```typescript
// Step 2: 有文件已上传且预览返回了列信息
store.inputs.some(inp => inp.fileId && store.uploadedFiles[inp.fileId]?.columns?.length)
// 提示: "AI 已分析列结构，点击「AI 分析列」查看详情"

// Step 3: 有输入源但 SQL 未填写
store.inputs.length > 0 && !store.processor.sql.trim()
// 提示: "描述你的查询需求，AI 帮你生成 SQL"

// Step 4: SQL 已填写但列映射为空
store.processor.sql.trim() && !(store.output?.config as any)?.columns?.length
// 提示: "AI 可自动推断列映射关系"

// Step 5: YAML 已生成
yamlPreviewRef?.yamlText
// 提示: "配置就绪，可以下载或执行"
```

- AiInlineTip 改动用 `suggestion` slot 传入自定义文案，保持组件简洁

### 引擎

#### E1: 多处理器链式执行

- **现状**: `engine.py:53` 只执行 `processors[:1]`，忽略第二个及之后的处理器
- **目标**: 遍历全部 processor，实现链式处理（后处理器可查询前处理器创建的表）
- **文件**: `src/pipeforge/core/engine.py`
- **测试文件**: `tests/test_engine_multiproc.py`
- **改动**: 去掉 `[:1]` 切片限制，改为遍历 `self.config.processors`
- **错误处理**: 保持当前策略 — 任一处理器失败则整个流水线失败，但失败的第一个处理器之前的 stats 已写入 `context.result.processors`，可在异常信息中查看已完成步骤
- **验收补充**: 第二个处理器失败时，第一个处理器 stats 仍在 `context.result` 中可查看

#### E2: 执行日志持久化

- **现状**: Logger 只写 stdout，执行完日志丢失
- **目标**: 支持写日志文件，失败时与 .db 路径一起输出，自动清理过期日志
- **文件**: `src/pipeforge/core/engine.py` 或新建 `src/pipeforge/utils/logging.py`
- **方案**:
  - Logger 接受 `log_dir` 参数，有值时同时写文件（`pipeforge_{timestamp}.log`），纯文本格式，含时间戳前缀
  - CLI 加 `--log-dir` 选项
  - Web API 默认日志目录：`configforge/tmp/logs/`
- **清理策略**: 自动删除 7 天前的日志文件，与 `files.py` 中 `cleanup_old_files()` 模式一致，在 `cleanup_old_files()` 旁边或内部实现 `cleanup_old_logs()`
- **日志格式**:
  ```
  [2026-05-19 15:30:01] [INFO] Input '人员明细': loaded 1500 rows in 0.23s
  [2026-05-19 15:30:02] [INFO] Processor '数据合并': SQL executed in 0.45s
  [2026-05-19 15:30:02] [INFO] Output: written 1200 rows to ./output/report.xlsx in 0.31s
  ```

## 验收标准

1. 5 个步骤卡片的图标区在浅色/暗色模式下各显示不同配色
2. Step2-5 在对应条件下出现 AI 内联提示条
3. `processors` 列表中的多个处理器依次执行，后处理器可查询前处理器创建的表
4. 执行后日志文件存在，含完整时间线（纯文本格式，时间戳前缀）
5. 7 天前的日志文件自动清理
6. `npx vue-tsc --noEmit` 零错误
7. `npm run build` 成功
8. `python3 -m pytest tests/ configforge/tests/ -v` 全部通过（含新增 `test_engine_multiproc.py`）
