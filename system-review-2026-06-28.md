## 1. 审查概要

CCTEST 是一个数据处理流水线框架，由三个核心模块组成：

| 模块 | 职责 | 技术栈 |
|------|------|--------|
| **PipeForge** | CLI 数据管道引擎 | Python 3.13, Pydantic v2, SQLite, openpyxl, Jinja2 |
| **ConfigForge** | Web API 服务 + AI 编排 | FastAPI, SQLAlchemy, OpenAI/Anthropic SDK |
| **ConfigForge Web** | 前端向导 + 配置管理 | Vue 3, TypeScript, Naive UI, CodeMirror 6, Pinia, Vite |

**核心价值：** 让不懂代码的人也能创建可重复的数据处理流程。

**当前状态：** Phase 1-3 已完成，Phase 4（生产就绪）待启动。

## 2. 系统架构分析

### 2.1 架构设计评价

**评分：92/100** ✅

**架构亮点：**

1. **清晰的三层架构**
   ```
   ConfigForge Web (Vue 3) → ConfigForge (FastAPI) → PipeForge (Python 引擎)
   ```
   - 职责分离明确：前端负责交互，API 负责编排，引擎负责执行
   - 依赖方向正确：`API → Service → Storage`，无反向依赖
   - 可独立部署：PipeForge 可脱离 Web 独立运行

2. **插件化设计**
   - 使用注册模式（Registry Pattern）+ 策略模式（Strategy Pattern）
   - 插件基类设计合理：`Plugin` 泛型基类 + `InputPlugin` / `ProcessorPlugin` / `OutputPlugin` 子类型
   - 扩展性强：新增输入源/处理器/输出格式只需实现对应插件接口

3. **服务层抽象**
   - `ConfigService`：封装配置管理的业务逻辑（校验、版本管理、审计）
   - `ExecutionService`：统一流水线执行逻辑（SSE 进度、失败处理、通知分发）
   - `ExecutionStore`：从 API 层抽离的执行记录存储（支持跨 worker 去重）

4. **依赖注入**
   - `ConfigService` 注入 `ConfigStoreProtocol`（存储后端可切换）
   - `ExecutionService` 注入 `ExecutionStore`（执行记录存储可替换）
   - 便于测试和存储后端迁移

5. **DAG 拓扑排序**
   - 多步 Pipeline 支持步骤间依赖（`input_tables` / `output_tables`）
   - 使用 Kahn's algorithm 进行拓扑排序
   - 自动检测循环依赖

**架构问题：**

1. **Windows 兼容性**
   - `signal.SIGALRM` 在 Windows 上不可用（已知限制，用户决策不修复）
   - 影响：Windows 上无超时保护（功能缺失，非崩溃）

2. **API 版本管理缺失**
   - 当前无 API 版本控制（v1 隐含在路由中）
   - 未来变更可能导致破坏性变更

3. **大模型供应商锁定风险**
   - 当前仅支持 OpenAI/Anthropic
   - 建议：抽象 `LlmBackend` 接口，支持更多供应商

## 4. 安全性分析

### 4.1 安全特性

**评分：95/100** ✅

**已实施的安全措施：**

1. **SQL 注入防护**
   - 输入源使用参数化查询（`SQLiteManager.execute()`）
   - 输出阶段 `safe_identifier()` 校验表名/列名
   - SQL 白名单：`SELECT-only` 模式（配置阶段校验）

2. **路径遍历防护**
   - `validate_id()` 校验所有文件路径参数
   - 禁止 `..`、`/`、`\` 等危险字符
   - 正则白名单：`^[a-zA-Z0-9_.-]+$`

3. **密码加密存储**
   - 数据库连接密码使用 Fernet 加密
   - 密钥通过环境变量 `CONFIGFORGE_ENCRYPTION_KEY` 配置
   - API 返回时自动脱敏（`sanitize_connection_string()`）

4. **SSRF 防护**
   - `validate_url()` 校验外部 URL
   - 禁止内部 IP（10.x, 172.16-31.x, 192.168.x, 127.x）
   - 禁止云元数据端点（169.254.169.254）

5. **文件上传安全**
   - 流式写入（避免大文件 OOM）
   - 文件锁（`fcntl.flock` 防止并发读写竞态）
   - 文件大小限制（50MB，nginx `client_max_body_size`）

6. **API 认证**
   - JWT 认证中间件（`AuthMiddleware`）
   - 生产模式强制 `CONFIGFORGE_JWT_SECRET`（长度 ≥ 32）
   - 可选 API Key 认证

7. **CORS 配置**
   - 生产模式强制 `CORS_ORIGINS` 配置
   - 禁止 `*` 通配符
   - 开发模式自动 fallback 到 localhost

8. **生产环境检查**
   - `lifespan()` 启动时检查关键配置
   - 缺失则 `raise SystemExit(1)`

**安全改进建议：**

1. **速率限制**
   - 当前有 `test_rate_limit.py`，但未在 `server.py` 中启用
   - **建议**：添加 `slowapi` 或 `fastapi-limiter`

2. **HTTPS 强制**
   - nginx 配置中 443 端口注释掉
   - **建议**：生产环境强制 HTTPS（HSTS 头）

3. **安全头**
   - nginx 已配置 `X-Frame-Options`、`X-Content-Type-Options` 等
   - **建议**：添加 `Content-Security-Policy`

4. **依赖漏洞扫描**
   - **建议**：集成 `pip-audit` 或 `safety` 到 CI/CD

## 5. 性能分析

### 5.1 性能特性

**评分：85/100** ✅

**性能优化措施：**

1. **流式处理**
   - 输入插件使用迭代器（`Iterator[tuple]`）
   - 避免一次性加载大文件到内存

2. **SQLite 临时库**
   - 每次执行新建临时 `.db`
   - 失败后保留用于调试

3. **openpyxl 两阶段写入**
   - 阶段 1（普通模式）：加载模板 → 提取样式
   - 阶段 2（write_only）：写入数据（高性能）
   - 平衡样式保留与写入性能

4. **异步执行**
   - FastAPI 异步路由
   - `asyncio.to_thread()` 将 CPU 密集型任务放到线程池

5. **连接池**
   - SQLAlchemy 连接池配置（未在代码中看到，建议添加）

6. **缓存层**
   - `utils/cache.py` 存在（功能未知，建议审查）

**性能问题：**

1. **无分页**
   - 预览接口 `/api/preview` 可能返回大量数据
   - **建议**：添加分页参数（`limit` / `offset`）

2. **无查询优化**
   - SQLite 未配置 WAL 模式（建议启用）
   - 未创建索引（输入数据量大时影响性能）

3. **前端打包体积**
   - `package.json` 中依赖较多（Naive UI 全量引入？）
   - **建议**：使用 `vite-plugin-imp` 按需引入

4. **无 CDN**
   - 静态资源未配置 CDN
   - **建议**：生产环境使用 CDN 加速

   - 当前仅支持 OpenAI/Anthropic
   - **建议**：抽象 `LlmBackend` 接口，支持更多供应商（如本地模型）

### 9.2 代码质量

1. **超长文件** 📄
   - `api/ai.py`：460 行
   - `services/execution_service.py`：540 行
   - **建议**：继续拆分子模块

2. **魔法字符串** 🔤
   - 部分错误提示硬编码中文
   - **建议**：抽取到常量或国际化文件
