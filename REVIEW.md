# PipeForge 设计文档 Review

> 审核文档: DESIGN.md (v0.1)  
> 审核日期: 2026-05-03

---

## 整体评价

文档结构完整，核心思路（三段式流水线 + SQL 处理 + Excel 模板输出）是务实且可落地的。但在需求定义、架构细节、异常处理等方面存在一些值得注意的问题。

---

## 一、需求层面

### 1.1 「AI 驱动」与「独立运行」的定位矛盾

文档第 2 节的核心用户旅程完全依赖 AI 交互（"AI 问" / "AI：请上传..."），但决策记录 #2 明确说「独立于 AI 工具运行」。这是一个根本性的定位问题：

- 如果 PipeForge 是 **AI 辅助创建配置 + 独立 CLI 执行**，那文档里应该把这两个阶段分开描述，而不是混在同一个用户旅程里。
- 「后续使用」中的 `AI：请上传本月人员明细 Excel` 其实和 AI 无关——这就是普通的参数提示，CLI 完全可以做到。

**建议**：明确区分「配置创建（可借助 AI）」和「配置执行（纯 CLI）」两个阶段，让用户旅程更精准。

### 1.2 缺少目标用户画像

决策 #10 说「使用者假设有基础编程技能，可以编写 SQL，理解 YAML 配置」。但这与第 2 节中 AI 辅助降低门槛的愿景矛盾——如果用户能写 SQL，为什么还需要 AI 来生成配置？

**建议**：明确两阶段的目标用户：
- **配置创建者**：需要能写 SQL、理解数据映射，可能是数据分析师或开发者
- **配置执行者**：不需要任何技能，只需根据提示提供输入文件

### 1.3 缺少非功能性需求

文档完全没有涉及：
- **数据规模上限**：最大支持多少行？1000 行？10 万行？100 万行？
- **性能预期**：处理 1 万行数据期望在多少秒内完成？
- **错误恢复**：执行到一半失败了怎么办？有没有断点重跑？
- **安全性**：SQLite 临时文件包含原始数据，清理失败时数据是否会泄露？

## 二、架构设计

### 2.1 责任链模式与实际功能不匹配

第 3.2 节列出了 **责任链模式**（多个 Processor 串联执行），但 MVP 版本只支持单处理器。多处理器链式执行被放在 v0.2+ 计划中。这意味着当下架构并为责任链预留扩展点，到时候可能需要重构。

**建议**：要么 MVP 就把 Processor 设计为列表（即使只有一个），要么不在设计模式中列出责任链，改为在 v0.2 时引入。

### 2.2 SQLite 临时库的表名冲突问题

多个 Input 写入同一个 SQLite 的临时表，但没有任何命名空间/前缀机制。如果两个 Excel 源的 sheet 恰好有相同的列名，或者用户在 SQL 中引用的表名和其他 Input 的 table 同名，就会出问题。

**建议**：
- 在 SQLite 管理器层面加表名前缀或命名空间
- 在配置加载时做表名冲突检测

### 2.3 Plugin 基类的 execute 签名设计有问题

```python
def execute(self, context: Context, config: dict) -> None:
```

- `config: dict` 失去了类型安全，配置由 Pydantic 校验完了到这里又是裸 dict？
- 每个插件的 config 结构不同，应该用泛型或至少让插件声明自己的 config schema

**建议**：让每个插件声明自己的 Pydantic model，由引擎在调用前做类型校验和注入。

### 2.4 缺少错误处理策略

没有任何地方描述：
- Input 读取失败怎么处理？
- SQL 执行出错怎么处理？
- Output 写入失败怎么处理？
- 事务回滚机制是什么？

**建议**：增加一个「错误处理」章节，至少覆盖：
- 各阶段错误分类（可恢复 / 不可恢复）
- 失败后的清理策略（临时 .db 是否保留以便调试？）
- 是否支持事务回滚到 Pipeline 执行前的状态

### 2.5 Context 中的 result 字段设计过于简单

`result: dict` 用来记录执行结果统计，但没有任何结构定义。三个阶段的统计维度不同（input 统计导入行数、processor 统计影响行数、output 统计导出行数），需要一个结构化的 result 模型。

## 三、配置设计

### 3.1 Template + Columns 映射的健壮性不足

Output 通过 `template` + `columns` 来定义输出列。问题在于：
- 如果 `columns` 里的列名在 `source_table` 中不存在怎么办？
- 如果 template 里的 sheet 名称不匹配怎么办？
- 只靠列名匹配，如果 template 的列头和 columns 不完全一致怎么办？

**建议**：
- 明确列缺失时的行为（报错 / 警告 / 跳过）
- 考虑增加列映射配置：`{template_header: source_column}` 的显式映射

### 3.2 SQL 注入风险

Processor 中的 SQL 是用户写在 YAML 里的，目前是直接执行的。虽然这是设计意图（用户自己写 SQL），但如果未来加上 Jinja2 变量替换（`{{param}}`），又没有参数化处理，就会有 SQL 注入风险。

**建议**：在引入 Jinja2 变量替换时，区分「安全变量」（日期、环境变量等系统变量）和「用户输入变量」，对后者做转义。

### 3.3 配置缺少版本兼容策略

`scene.version` 字段存在，但没有描述版本升级时如何处理旧配置。如果 v1.0 的配置格式在 v2.0 中不兼容怎么办？

**建议**：增加配置迁移策略说明，或者至少在 Pydantic model 中支持宽松解析（忽略未知字段而非报错）。

## 四、技术选型

### 4.1 pandas 依赖是否过重？

pandas 是一个 ~100MB 的依赖，仅用来做「读 Excel → 写 SQLite」。实际上 `openpyxl` 可以直接逐行读取，然后通过 `sqlite3` 的 `executemany` 写入，完全不需要 pandas。这也避免了 pandas 将整个 Excel 加载到内存的问题。

**建议**：评估是否真的需要 pandas。如果只是为了 `DataFrame.to_sql()` 这个便利，代价过高。

### 4.2 Jinja2 的使用场景不清晰

技术选型中引入了 Jinja2，但 MVP 明确说「SQL 中是否需要变量替换暂不需要」。Jinja2 是 v0.2 的功能，不应出现在当前版本的技术选型表中（否则容易让人误以为 MVP 就需要）。

---

## 五、文档结构

### 5.1 章节编号跳跃

文档从第 9 节（决策记录）直接跳到第 11 节（实现计划），缺少第 10 节。

### 5.2 第 12 节「待明确事项」应该前置

这些问题（变量替换、列映射方式、temp 文件路径、处理器依赖）是核心设计决策，不应该作为「待明确」留到最后。建议在实现前就把这些问题定下来。

---

## 六、总结建议优先级

| 优先级 | 问题 | 建议 |
|--------|------|------|
| **P0** | AI 定位模糊 | 明确区分「配置创建」和「配置执行」两个阶段 |
| **P0** | 缺少错误处理策略 | 增加错误处理章节，定义各阶段失败的行为 |
| **P0** | 责任链模式与 MVP 不一致 | 移除或预留扩展点 |
| **P1** | pandas 依赖评估 | 确认是否真的需要，考虑纯 openpyxl 方案 |
| **P1** | 列映射健壮性 | 定义列缺失行为，考虑显式映射 |
| **P1** | 表名冲突 | 增加命名空间或冲突检测 |
| **P2** | Plugin config 类型安全 | 用泛型 Pydantic model 替代裸 dict |
| **P2** | 待明确事项后置 | 提到设计阶段早期决策 |
| **P2** | 章节编号缺失 | 补充第 10 节或重新编号 |

---

整体来说，这是一个方向正确的设计，核心架构（Input → SQLite → SQL → Output）是务实可行的。主要风险集中在 **AI 定位的模糊** 和 **异常路径缺乏设计** 这两块，建议优先解决。
